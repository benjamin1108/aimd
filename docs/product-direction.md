# AIMD Product Direction

## 核心判断

AIMD 现在的基础能力是把 Markdown、图片和元数据打包进一个 `.aimd` 单文件容器。这是一个合理的技术底座，但单独作为产品卖点还不够尖锐。

用户不关心“新文档格式”本身。用户关心的是：AI 生成的内容能不能漂亮、完整、可靠地保存、交付和再编辑。

因此，AIMD 不应该先做成一个大而全的 Markdown 编辑器，而应该聚焦在：

> AI 时代的可交付文档容器：一份文件里包含正文、图片、引用资源、生成元数据和版本信息，打开就能读，交付不会丢图。

## 产品定位

AIMD 可以定位为 AI 生成内容的交付和归档工具。

更具体的一句话：

> AIMD 把 AI 生成的 Markdown、图片和上下文打包成一份可离线、可归档、可交付的完整文档。

这个定位比“Markdown 编辑器”更有差异化。Markdown 编辑器已经很多，但 AI 报告、Agent 输出、调研资料、图文方案如何可靠交付，还没有一个轻量、开放、离线友好的标准工具。

## 目标用户

- 经常用 AI 生成日报、调研报告、竞品分析、技术方案的人
- 需要把 ChatGPT、Claude、Codex、Agent 输出整理成可交付材料的人
- 需要保存截图、图表、引用资料和 Markdown 正文的技术/产品/运营用户
- 想把 AI 产物归档，但不想维护 Markdown 文件夹、图片目录和外链的人

## 关键方向

### 1. AI 报告交付

让用户把 AI 生成的报告、图片、图表、表格和附件打包成一个 `.aimd` 文件。

重点不是替代所有编辑器，而是保证交付物完整：

- 图片不丢
- 表格和代码块可读
- 文档离线可打开
- 文件可以直接发给同事、客户或归档
- 后续仍然可以编辑，而不是像 PDF 一样变成终点格式

### 2. 生成过程溯源

`.aimd` 不只保存正文，也可以保存 AI 生成过程的上下文。

可以记录：

- 使用的模型
- 原始 prompt
- 生成时间
- 内容来源和引用
- 人工修改记录
- 审阅状态

这会让 AIMD 从“文档文件”升级为“AI 产物证据包”。这对企业内部报告、研究资料、Agent 产物尤其有价值。

### 3. 一键导入 AI 输出

导入体验应该是核心能力。

优先场景：

- 从 ChatGPT/Claude 页面复制内容，粘贴后自动整理成干净文档
- 从 Markdown 文件导入并自动内联本地图片
- 从 HTML 表格、截图、富文本粘贴中恢复结构
- 从 Agent 运行结果生成 `.aimd`

用户不应该从空白页开始写 AIMD。AIMD 应该把已有 AI 输出快速整理成可交付文档。

### 4. 多格式导出

`.aimd` 可以作为母版格式，但最终交付对象不一定要使用 `.aimd`。

需要支持导出：

- PDF
- 单页 HTML
- Markdown + assets 文件夹
- DOCX
- ZIP 交付包

这样 AIMD 的角色会更清晰：它是可编辑、可归档的源文件，导出格式是面向不同场景的交付结果。

### 5. 文档健康检查

面向交付的工具需要能告诉用户这份文档是否可靠。

可以检查：

- 是否有缺失图片
- 是否有未引用资源
- 是否有过大图片
- 是否有失效链接
- 是否可以离线打开
- 是否包含外部隐私链接
- 是否适合发给客户或归档

这类能力会让 AIMD 有专业感，也能和普通 Markdown 编辑器拉开差异。

### 6. 模板化报告

不要只提供空白编辑器。

可以内置高频 AI 交付模板：

- AI 调研报告
- 技术方案
- 产品需求文档
- 周报
- 会议纪要
- 竞品分析
- 数据分析简报
- Agent 执行报告

用户打开 AIMD 时，第一感受应该是“我可以马上产出一份交付物”，而不是“这是另一个 Markdown 编辑器”。

## 应避免的方向

- 不要和 Typora、Obsidian、Notion 正面对打
- 不要过早做知识库、同步、多端协作
- 不要把重点放在“新格式”宣传上
- 不要把编辑器功能堆成主要卖点
- 不要让用户从空白页开始理解产品价值

## 下一阶段优先级

### P0：导入体验

目标：把 AI 输出快速变成 AIMD。

- 粘贴 ChatGPT/Claude 富文本并保持结构
- 导入 Markdown 文件夹并打包图片
- 粘贴截图自动成为内联资产
- 清理 AI 输出中的多余样式和噪音

### P1：导出体验

目标：让 AIMD 能自然进入现有工作流。

- 导出 PDF
- 导出单页 HTML
- 导出 Markdown + assets
- 保持图片、表格、代码块和标题结构稳定

### P2：交付体验

目标：让用户相信这份文档可以发出去。

- 封面和基础元数据
- 文档健康检查
- 资源体积优化
- 断链和缺图提醒
- 漂亮、稳定的阅读模式

## 成功标准

AIMD 的吸引力不应该来自“它支持 Markdown”，而应该来自这些用户感知：

- 我从 AI 复制出来的内容，一下子变成了干净文档
- 我发出去的是一个文件，不会丢图
- 我以后还能继续编辑，不像 PDF 那样封死
- 我知道这份文档用了什么模型、什么 prompt、哪些来源
- 我可以按需要导出成 PDF、HTML 或 Markdown 文件夹

如果这些体验成立，AIMD 就不只是一个桌面 Markdown 壳，而是一个面向 AI 内容交付的工具。
