# AIMD 桌面版 QA 报告

> 本轮 QA 结论：当前主线可编译，核心静态检查通过，但体验上还有几个必须修的阻塞点：移动端布局严重异常、关闭文档入口被藏导致 16 个 E2E 失败、Docu-Tour/LiteLLM 的生产化依赖仍不稳、菜单和导览状态反馈还不够"桌面应用"。

## 测试结果

- `npm run typecheck`：通过
- `npm run build:web`：通过
- `cargo check`：通过
- `cargo test`：通过
- `npx playwright test`：216 通过，16 失败
- 视觉截图已生成：`/tmp/aimd-qa-*.png`

## P0 阻塞问题

### P0-1 关闭文档被藏进 ⋯ 菜单后，现有 E2E 全部点不到 `#close`

- **现象**：16 个失败都集中在这里。
- **位置**：`apps/desktop/src/ui/template.ts:97`
- **影响**：不只是测试问题，也是 IA 问题——关闭/丢弃这类高风险动作不能没有清晰状态、分隔线和可发现性。
- **修复方向**：让"关闭文档"在 ⋯ 菜单里仍保留可访问（保留 `id=close` 或同等可定位元素），同时 e2e 用菜单展开后再点击的路径定位它。

### P0-2 移动端 / 窄窗口布局崩坏

- **现象**：截图里顶部区域出现约 300px 空白，正文被严重挤下去。
- **位置**：`apps/desktop/src/styles.css:1589`
- **修复方向**：审视窄屏 media query，里面应当有错误的 `padding-top` / `margin-top` 或者绝对定位高度。

### P0-3 LiteLLM 仍依赖系统 `python3 + pip install litellm`

- **现象**：DMG 用户体验不合格，安装后大概率继续遇到 `No module named litellm`。
- **位置**：`apps/desktop/src-tauri/src/lib.rs:463`
- **修复方向**：本轮先做"运行前检测 + 友好错误提示 + 一键复制 pip 命令"的兜底，长期方案另议（打包 sidecar Python 或绕过 LiteLLM SDK 直接调 DashScope / Gemini HTTP API）。

### P0-4 `lib.rs` 已经过长

- **现象**：菜单、命令、Docu-Tour 进程调用都挤在一起，和"拒绝超长文件 / 模块合理拆分"的目标不一致。
- **位置**：`apps/desktop/src-tauri/src/lib.rs:819`
- **修复方向**：把 Docu-Tour 相关命令、菜单/事件绑定拆出独立模块（如 `docutour.rs`、`menu.rs`），保持 `lib.rs` 只做装配。

## P1 体验问题

### P1-1 ⋯ 菜单里"关闭文档"危险样式 / 分组缺失

- **现象**：没有危险动作样式，也没有和普通操作分组，风险感不足。
- **修复方向**：菜单中独立分组 + 危险色样式 + 与"打开 / 最近使用"等动作分隔。

### P1-2 "导览"菜单信息层级偏弱

- **现象**：播放导读、生成 / 更新导览权重接近，但没有显示当前文档是否已有导览、是否过期、是否正在生成。
- **修复方向**：在菜单项上显示状态徽标 / 副标题，例如"无导览 · 点击生成"、"已存在 · 上次生成于…"。

### P1-3 播放导读的控制条 / 关闭按钮 / 聚光框

- **现象**：底部 bar 会压住文档底部状态区；右上角关闭 × 与顶部 ⋯/保存区视觉打架；聚光框黑边过重偏调试态。
- **修复方向**：底部 bar 控制 z-index 与下边距；关闭按钮远离顶部 ⋯ 区域或合并；聚光框改半透明遮罩 + 柔和 outline。

### P1-4 设置页缺少完整配置体验

- **现象**：没有"测试连接"、没有 LiteLLM 安装状态、没有 provider 必填校验、API Base 的"可选"说明太弱。
- **修复方向**：补"测试连接"按钮 + 必填校验 + 状态条显示 LiteLLM 是否就绪。

### P1-5 API Key 存在 localStorage，安全性不够

- **位置**：`apps/desktop/src/docutour/config.ts:25`
- **修复方向**：桌面端走系统 Keychain / secure store（如 `tauri-plugin-stronghold` 或 keyring）。本轮可先标注 TODO + 在 UI 上提示"密钥保存在本地浏览器存储"。

### P1-6 Debug Console 全局 monkey patch console

- **位置**：`apps/desktop/src/debug/console.ts:55`
- **现象**：能力有用但生产态风险偏高。
- **修复方向**：变成显式 Debug Mode，默认关闭，菜单里打开。

## P2 打磨问题

### P2-1 阅读/编辑/源码切换图标与文字混排不统一

- **现象**：像临时工具栏。
- **修复方向**：统一图标尺寸 / 文字字号 / 间距，或者纯文字 / 纯图标二选一。

### P2-2 顶部保存按钮和 ⋯ 菜单视觉权重接近

- **现象**：右上角仍略拥挤。
- **修复方向**：保存按钮主色化，⋯ 菜单图标化降权。

### P2-3 Source 模式仍会看到 YAML / base64 元数据

- **现象**：渲染态已隐藏，但源码态体验仍会刺眼。
- **修复方向**：明确产品策略——源码忠实显示，还是折叠 / 虚化 Docu-Tour metadata。本轮可先做"折叠区"展示。

### P2-4 菜单命令通过字符串事件和前端 if/else 分发

- **位置**：`apps/desktop/src/main.ts:189`
- **现象**：后续命令多了会脆。
- **修复方向**：建立 command registry / map，按 id 注册回调。

## 下一轮建议顺序

1. 先修移动端 header 空白和关闭文档入口 / E2E 失败。
2. 重做菜单分组：File / View / Tools 保持，但危险动作、导览动作、调试动作要有明确层级和状态。
3. 把 Docu-Tour 后端从 `lib.rs` 拆出去，同时解决 LiteLLM 生产依赖方案。
4. 增加设置页"保存成功/失败/测试连接/依赖状态"完整闭环。
5. 再做导览播放态视觉打磨：控制 bar、关闭入口、聚光框、生成状态、手动切换体验。
