## 1. `.aimd` 文件图标

### 问题

- 当前 `.aimd` 文件已经有关联，但文件图标体验不完整
- 用户在 Finder / 桌面 / 打开对话框里，无法一眼识别 `.aimd`
- 应用图标、文件图标、文档类型视觉语言可能还未统一

### 目标

- `.aimd` 文件在 macOS 上显示明确、可识别的专属图标
- 图标风格与 `AIMD Desktop` 应用本体保持一致
- 安装后的文件关联不仅可打开，还能正确展示文档类型

### 要做

- 梳理当前 Tauri bundle 与 macOS file association 配置
- 确认文件类型声明是否已完整覆盖图标链路
- 设计并产出 `.aimd` 文件图标资源
- 接入打包配置，验证 Finder / 桌面 / “打开方式” 场景
- 检查已生成 `.app`、`.dmg` 安装后的真实表现

### 验收标准

- 双击 `.aimd` 可由 `AIMD Desktop` 打开
- Finder 中 `.aimd` 显示专属文件图标
- 图标不是默认白纸，也不是应用图标错误复用
- 新装环境和已有环境都能稳定识别

### 优先级

- `P1`

## 2. 图片压缩

### 问题

- 剪贴板粘贴图片时，文件体积可能显著膨胀
- 当前已去掉运行时 `base64/data:` 方案，但落盘资源仍可能过大
- 用户预期是“粘贴即用”，不希望一个中等图片把 `.aimd` 撑到异常大小

### 目标

- 控制粘贴/插入图片后的 `.aimd` 体积增长
- 优先降低剪贴板图片导致的 PNG 膨胀问题
- 在体积、清晰度、速度之间做明确权衡

### 要做

- 区分两类来源：
  - 文件选择插入
  - 剪贴板粘贴插入
- 查明当前粘贴链路拿到的图片编码格式与尺寸信息
- 定义压缩策略：
  - 是否限制最大边长
  - 是否重编码为 JPEG / WebP / PNG
  - 是否按阈值触发压缩
- 定义用户体验：
  - 静默压缩
  - 或大图给出提示
- 加入真实回归测试：
  - 粘贴大图后可正常打开
  - 文件体积明显低于未压缩方案
  - 保存后仍保持 `asset://` 语义

### 验收标准

- 粘贴大图后 `.aimd` 不再出现异常体积膨胀
- 打开、保存、重开都稳定
- 画质损失在可接受范围内
- 不回退到 `base64` / `data:` 方案

### 优先级

- `P0`

---

## 执行顺序

1. 先做图片压缩
2. 再做 `.aimd` 文件图标

原因：

- 图片压缩直接影响文件可用性和体积
- 文件图标更偏交付完善，但不阻塞核心可用性
