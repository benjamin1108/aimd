# /goal: 一键格式化的质量判断与温和清洗优化

## 背景

AIMD 已有“一键格式化”能力，但当前实现和 prompt 容易把格式化理解成硬性的结构重写，例如必须有 H1、必须重排标题层级等。

这不符合产品预期。一键格式化的核心价值不是把所有文档强行改成同一种模板，而是判断当前文档是否真的需要整理，并在需要时做温和、可靠的清洗与排版优化。

用户实际需求：

- 当前文档如果已经工整，就不要为了满足机械规则而重写。
- 是否需要格式化应交给大模型基于整体观感判断，而不是用“有没有 H1”“H1 是否唯一”这类死规则。
- 主要优化对象是杂乱复制内容：奇怪字符、非预期噪声、无意义页眉页脚、复制残留、重复空白、断裂段落、混乱列表、明显的广告/导航残片等。
- 不应删除关键内容，不应强制生成新的事实，不应把好文档改坏。

## 产品目标

把“一键格式化”优化成一个质量判断驱动的流程：

1. 先判断当前文档是否需要格式化。
2. 如果不需要，给出轻量状态提示，不打开预览、不覆盖文档。
3. 如果需要，进行温和格式化和清洗。
4. 清洗重点是提升可读性、去掉非预期噪声、保留原文关键信息，而不是强制套标题模板。

## 非目标

- 不做后台自动格式化；仍由用户主动点击“一键格式化”。
- 不引入复杂的模板系统。
- 不把格式化变成事实改写、总结改写或内容创作。
- 不强制所有文档必须有 H1/H2/H3。
- 不因为缺少 H1 就判定格式化失败。
- 不删除代码块、链接、图片、表格、列表、引用、命令、数字等关键内容。

## 需要改进的问题

### 当前过硬规则

检查并优化现有实现中类似这些过硬判断：

- prompt 中要求 body 必须包含 exactly one H1。
- prompt 中要求长文必须有 H2/H3。
- 前端 `validateFormattedMarkdown` 按 H1 数量硬失败。
- 任何把“格式化质量”简化成固定 Markdown 结构的判断。

这些规则可以作为模型参考，但不应作为一键格式化是否成功的硬门槛。

### 当前缺少“是否需要格式化”的判断

点击“一键格式化”后，不应默认一定产生一份重写结果。模型应先判断当前文档是否已经足够工整：

- 如果文档结构清晰、字符干净、没有明显复制噪声，应返回“不需要格式化”。
- UI 应提示类似：`当前文档已经比较工整，无需格式化`。
- 不应打开 diff/预览，也不应设置 dirty。

## 目标行为

### 判断标准

模型判断是否需要格式化时，应综合考虑：

- 文档是否整体工整、可读。
- 是否存在乱码、奇怪控制字符、异常符号、复制残留。
- 是否存在明显无关内容，例如网页导航、广告、登录提示、分享按钮文本、页眉页脚、版权噪声。
- 段落是否被复制过程打断，是否有异常换行。
- 列表、表格、引用、代码块是否明显破损。
- 标题、段落、列表是否有基础层次，但不要求固定标题结构。
- 是否存在大量重复空行、重复标点、重复文本片段。

不要用单一结构规则判断，例如：

- 缺少 H1 不等于需要格式化。
- 多个 H1 不一定需要格式化。
- 没有 H2/H3 不一定需要格式化。
- 短文、笔记、清单、代码说明可以没有完整章节结构。

### 格式化策略

当模型判断需要格式化时：

- 保留原文关键事实、链接、图片、代码、表格、命令和数字。
- 清理非预期的奇怪字符、控制字符、复制噪声。
- 删除明显非关键内容，例如无意义导航、广告残片、重复按钮文本。
- 修复明显断裂的段落和列表。
- 统一合理空行。
- 标题层级可以整理，但只在源文档确实有层次混乱时进行。
- 不为了生成“标准文章”而重写用户原意。
- 不把摘要或关键点强行塞进正文。
- YAML frontmatter 可以继续保存摘要、关键词、语言、格式化信息，但不要以生成 meta 为理由重写正文。

### 不需要格式化时

当模型判断不需要格式化时：

- 不返回一份和原文高度相似但细节不同的文档。
- 不打开预览确认面板。
- 不改 `state.doc.markdown`。
- 不设置 dirty。
- 给出明确但不打扰的状态提示。

建议状态：

```text
当前文档已经比较工整，无需格式化
```

## 实现要求

### 后端输出契约

不要只让模型输出裸 Markdown。需要让模型返回结构化结果，至少包含：

```json
{
  "needed": true,
  "reason": "文档存在复制噪声和异常换行",
  "markdown": "---\\ntitle: ...\\n---\\n\\n..."
}
```

或等价结构。

字段语义：

- `needed`: 是否需要格式化。
- `reason`: 简短原因，用于日志或调试，不需要强提示用户。
- `markdown`: 仅当 `needed=true` 时包含格式化后的完整 Markdown。

如果现有 `generate_text` 只方便返回文本，可以要求模型返回 JSON，并在 Rust 侧解析；解析失败时应安全失败，不覆盖原文。

### Prompt 改造

更新 `apps/desktop/src-tauri/src/prompts/format_markdown_system.md`：

- 明确第一步是判断是否需要格式化。
- 去掉“body must contain exactly one H1”这类硬性要求。
- 把 H1/H2/H3 改成“在确有必要时整理标题层级”。
- 强调工整度、噪声清理、温和修复。
- 明确不需要格式化时返回 `needed=false`。

语言策略文件继续保留，但不要强迫所有文档翻译或重写；如果输出语言与原文不同，仍需忠实保留完整语义。

### 前端流程

更新 `apps/desktop/src/document/format.ts`：

- `format_markdown` 返回结构化结果后，先检查 `needed`。
- `needed=false` 时：
  - 关闭/不打开预览。
  - 不改文档。
  - 不设置 dirty。
  - 显示轻量状态。
- `needed=true` 时：
  - 只对安全性做必要验证。
  - 不再用 H1 数量作为硬失败条件。
  - 继续检查链接、`asset://` 资源、正文非空等关键保护。

### 后端流程

更新 `apps/desktop/src-tauri/src/formatter.rs`：

- 返回结构化类型，而不是裸 `String`，或在 Tauri 命令边界返回可序列化对象。
- 解析模型 JSON 输出。
- 对 `needed=false` 做直通返回。
- 对 `needed=true` 做 Markdown fence 清理和基础校验。
- 模型返回异常、JSON 解析失败、缺少 markdown 时，返回错误，不覆盖用户文档。

## 安全与保护

必须保留这些硬保护：

- 输入为空时不格式化。
- 输出正文为空时失败。
- 原文中的 `asset://` 引用不能丢。
- 原文中的 Markdown 链接 URL 不能丢。
- 代码块不能被破坏。
- 表格、列表、命令应尽量保留。

可以移除或降级为 warning 的规则：

- H1 数量。
- 是否存在 H2/H3。
- 标题是否满足文章模板。

## 测试要求

### Rust 单元测试

新增或更新测试覆盖：

- prompt 包含 `needed` 判断要求。
- `needed=false` JSON 能正确解析并返回。
- `needed=true` JSON 能取出 markdown。
- markdown 被代码块 fence 包裹时能清理。
- JSON 解析失败时返回错误。

### TypeScript 单元或 E2E

覆盖：

- `needed=false` 时不打开预览、不设置 dirty。
- `needed=true` 时打开预览。
- 缺少 H1 不再导致格式化结果失败。
- 丢失 `asset://` 或链接仍然失败。

### 手动验证

至少验证三类文档：

1. 已经工整的短文或笔记：点击后提示无需格式化。
2. 从网页复制的杂乱内容：能清理导航/噪声/异常空行。
3. 没有 H1 的简单清单：不应因为没有 H1 而失败。

## 验收标准

- 一键格式化不再机械要求 H1/H2/H3。
- 模型会先判断当前文档是否需要格式化。
- 不需要格式化时不会打开预览、不会改文档、不会设置 dirty。
- 需要格式化时能清理明显噪声并保持关键信息。
- 链接、`asset://` 图片、代码块、表格等关键内容保护仍然有效。
- 解析失败或模型输出异常时保留原文。
- 相关测试通过。
