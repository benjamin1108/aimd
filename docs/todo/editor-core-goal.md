# /goal AIMD 编辑器核心能力补齐

## Objective

把 AIMD Desktop 从“能打开和轻量编辑 `.aimd` 的工具”补齐到“能稳定导入、编辑、检查、导出图文 Markdown 交付物”的编辑器。

只做四件事：

1. 导入增强
2. 导出增强
3. 资源可靠性检查
4. 基础编辑能力补齐

目标不是做知识库、协作平台、AI 溯源系统或大而全 Markdown 编辑器。每一项都必须直接服务“图片不丢、结构不乱、能交付、能继续编辑”。

## Non-Goals

- 不做 AI 溯源、AI 证据包、模型审计、prompt 认证。
- 不做实时协作、云同步、账号系统。
- 不做知识库、双链、图谱、插件市场。
- 不做模板中心，除非实现过程中需要极小的测试 fixture。
- 不要求在 Linux 环境打包 macOS DMG。
- 不为了功能堆叠引入复杂产品面板；优先保持当前桌面应用结构。

## Constraints

- 继续保持 `.aimd` 是开放 ZIP 容器：`manifest.json` + `main.md` + `assets/`。
- 普通 `.md` 仍然必须能原样打开和保存；只有当文档依赖内嵌资源时才转为 `.aimd`。
- 图片、表格、代码块是核心兼容对象，任何导入/导出改动都不能破坏这些结构。
- 所有新功能必须有可运行的测试，不能只做 UI 壳。
- Linux 开发环境下必须完成构建、类型检查、Rust 测试和可行的 Playwright/E2E 测试；最终 DMG 打包可以跳过。

## Current Baseline

当前已有能力：

- 阅读 / 编辑 / 源码三模式。
- `.aimd` 打开、保存、另存为。
- 普通 Markdown 打开和保存。
- 粘贴 / 插入图片并打包进 `.aimd`。
- 图片去重、大图压缩、保存时清理未引用资源。
- Web Clip 提取网页并生成未保存 `.aimd` 草稿。
- 大纲、资源面板、图片灯箱、多窗口、最近打开。

本 TODO 要补的是当前编辑器作为交付工具最明显的缺口。

## P0: 导入增强

### Goal

用户从 AI、网页、Markdown 文件夹拿到内容后，可以一键变成 `.aimd`，并且图片不丢、表格和代码块不乱。

### Required Work

- 支持“导入 Markdown 项目/文件夹”：
  - 选择一个 `.md` 文件或项目目录。
  - 扫描 Markdown 中的本地图片引用。
  - 将本地图片复制进 `.aimd` 的 `assets/`。
  - 将本地图片路径改写成 `asset://...`。
  - 保留远程 URL，不静默下载，除非用户明确选择本地化。
- 强化普通 `.md` 打开后的“转为 `.aimd`”路径：
  - 如果 Markdown 内已有本地图片引用，提供明确入口打包成本地资源。
  - 转换后正文引用和资源清单必须一致。
- 强化富文本 / AI 输出粘贴：
  - 尽量保留标题、列表、表格、代码块、链接。
  - 清理复制来源里的无关样式、按钮、导航、脚本、空标签。
  - 粘贴图片仍应进入 `.aimd` 草稿资源包。
- 强化 Web Clip 导入：
  - 表格、代码块、图片引用在 HTML -> Markdown -> AIMD 流程中保持稳定。
  - 失败时状态明确，不留下无法保存的半成品。

### Acceptance Criteria

- 导入带本地图片的 Markdown 后，保存出的 `.aimd` 能离线打开，图片全部显示。
- 导入含 GFM 表格的 Markdown 后，阅读模式和源码模式表格结构保持一致。
- 导入含 fenced code block 的 Markdown 后，语言标记不丢失。
- 富文本粘贴不会带入危险 HTML、脚本、内联事件。
- 导入失败不会污染最近打开列表，也不会留下被当前文档误引用的临时资源。

### Tests

- Rust 测试覆盖 Markdown 本地图片打包、路径改写、重复图片、缺失图片、远程图片保留。
- 前端或 E2E 测试覆盖：
  - 打开 `.md` 后打包成本地 `.aimd`。
  - 富文本粘贴表格。
  - 富文本粘贴代码块。
  - 粘贴图片后保存为 `.aimd`。
  - Web Clip 失败和重试路径。

## P1: 导出增强

### Goal

`.aimd` 是可编辑母版，但收件人不一定安装 AIMD。必须能导出成常见交付格式。

### Required Work

- 导出 PDF。
- 导出单页 HTML：
  - 图片必须可离线显示。
  - 样式应随文件自包含，或生成明确的交付包。
- 导出 Markdown + assets 文件夹：
  - `main.md` 中的 `asset://...` 改写为相对路径。
  - assets 文件夹只包含被正文引用的资源。
  - 文件名稳定、可读、避免冲突。
- 导出流程需要明确状态：
  - 成功导出到哪个路径。
  - 导出失败原因。
  - 用户取消不算错误。
- 交付格式入口应放在现有文档操作中，不引入复杂新工作区。

### Acceptance Criteria

- `.aimd` 导出 Markdown + assets 后，不依赖 AIMD 也能在普通 Markdown 查看器中看到本地图片。
- 导出 HTML 后，断网情况下图片仍可显示。
- 导出 PDF 后，标题、段落、表格、图片、代码块基本可读。
- 未引用资源不会被导出。
- 导出不会改变当前打开文档的 dirty 状态，除非用户同时修改了正文。

### Tests

- Rust 测试覆盖 `asset://` 到相对路径的导出改写。
- Rust 测试覆盖仅导出被引用资源。
- 前端或 E2E 测试覆盖导出菜单、取消、成功状态、失败状态。
- 至少有一个 fixture 同时包含图片、表格、代码块，用于导出回归。
- Linux 环境下可以跳过 macOS DMG 打包，但不能跳过导出功能测试。

## P2: 资源可靠性检查

### Goal

用户能在保存或导出前知道文档是否会丢图、是否还依赖外部图片、是否存在无用或过大的资源。

### Required Work

- 实现文档健康检查：
  - Markdown 中引用的 `asset://id` 是否存在于 manifest。
  - manifest 中是否有未被正文引用的资源。
  - 是否有远程图片 URL。
  - 是否有本地文件路径图片引用仍未打包。
  - 是否有过大图片。
  - 是否有无法识别的资源 MIME。
- 在 UI 中提供明确结果：
  - 可离线交付。
  - 有风险，但可保存。
  - 有缺失资源，应修复。
- 对可自动修复的问题提供操作：
  - 清理未引用资源。
  - 压缩过大图片。
  - 将本地图片打包进 `.aimd`。
  - 将远程图片本地化时必须由用户明确触发。
- 健康检查不应阻塞普通编辑，但导出前应提示高风险项。

### Acceptance Criteria

- 缺失 `asset://` 引用能被检测出来，并显示具体 id。
- 未引用资源能被检测出来，并可清理。
- 本地图片路径未打包能被检测出来。
- 远程图片能被检测出来，但不会静默下载。
- 大图片能被检测出来，并复用现有压缩逻辑或提供一致的新压缩入口。

### Tests

- Rust 单元测试覆盖健康检查核心逻辑。
- 前端或 E2E 测试覆盖：
  - 健康检查入口。
  - 缺图提示。
  - 未引用资源清理。
  - 远程图片提示。
  - 大图片提示。
- 保存和导出前风险提示要有回归测试。

## P3: 基础编辑能力补齐

### Goal

补齐用户每天会遇到的编辑摩擦，但不把 AIMD 做成大而全编辑器。

### Required Work

- 查找 / 替换：
  - 阅读、编辑、源码模式至少支持查找。
  - 源码模式支持替换。
  - 查找结果需要高亮或可跳转。
- 源码高亮：
  - 将原生 textarea 升级为专业 Markdown 编辑组件，或实现足够稳定的 Markdown 高亮。
  - 支持软换行、当前行、基础快捷键。
  - 不能破坏现有源码编辑、预览同步和图片粘贴。
- 表格编辑：
  - 插入基础 GFM 表格。
  - 在编辑模式中能添加 / 删除行列，或至少提供可用的表格编辑入口。
  - 从剪贴板粘贴表格时尽量保留结构。
- 代码块：
  - 插入 fenced code block。
  - 编辑语言标记。
  - 阅读模式中代码块可复制。
- 图片 alt / caption：
  - 能编辑图片 alt 文本。
  - 如实现 caption，必须稳定往返 Markdown，不破坏普通图片语法。
- 任务列表：
  - 阅读或编辑模式点击 checkbox 后同步 Markdown。

### Acceptance Criteria

- 查找 / 替换不会丢失当前文档修改。
- 源码高亮不会明显拖慢大文档输入。
- 表格在源码、编辑、阅读三种模式之间往返后不损坏。
- 代码块语言标记保存后仍存在。
- 图片 alt 修改后，正文 Markdown 同步更新。
- 任务列表 checkbox 点击后，`- [ ]` 和 `- [x]` 能正确互换。

### Tests

- 前端或 E2E 测试覆盖查找、替换、源码编辑同步。
- E2E 测试覆盖表格插入和模式切换。
- E2E 测试覆盖代码块语言编辑。
- E2E 测试覆盖图片 alt 修改。
- E2E 测试覆盖任务列表点击同步 Markdown。
- 性能回归至少覆盖一个较长 Markdown fixture，确认源码编辑不会明显卡顿。

## Required Verification

完成实现后必须运行：

```bash
cd apps/desktop
npm run typecheck
npm run build:web
npm run test:e2e
```

```bash
cargo test
```

如果某个测试因为 Linux 环境限制无法运行，必须在最终说明中写清楚：

- 哪个测试没跑。
- 为什么没跑。
- 用什么替代验证。

可以不做：

```bash
./build-dmg.sh
```

## Definition of Done

- 四个方向中本次选择实现的功能都有真实代码，不只是 UI 占位。
- 每个新增用户可见功能都有测试。
- 普通 `.md` 工作流不被破坏。
- `.aimd` 保存、另存为、图片粘贴、Web Clip 现有路径不回退。
- 文档、状态提示、失败提示和测试结果都更新到位。
- Linux 环境完成开发验证；macOS DMG 打包明确跳过。
